2. Vlan-ы или мультиплексирование.
	Задача:	Как через один провод передавать данные 2-х и более различных потоков.
	Решение:	Для оптического кабеля (технология WDM)-на одной частоте передавать 1-ый поток, на второй частоте 2-ой поток
			В мобильной связи (TDM)-в первую секунду передаёт 1-ый поток, во вторую сек 2-ой, в третью опять 1-ый, в четвертую - 2-ой, 5 - 1 и т.д.
			В Ethernet (VLAN)-каждый пакет маркируем меткой = Vlan ID. В зависимости от Vlan ID понимаем, к какому потоку относится пакет.

Пакеты на уровне МАС-адресов бывают тольк двух типов - Untagged и Tagged

Untagged:

|  6 байт	|  6 байт	|2байт|		|  4	|
|-----------------------------------------------|
|MAC Destin	|MAC Source	|Type	|  Payload	| FCS	|

значение поля Type определяет какой Layer 3 заголовок будет дальше в пакете
0x0800 - IP v4
0x86DD - IPv6
0x0806 - ARP
0x8100 - Vlan 802.1q
0x8137 - IPX

Tagged, внутрь вставляется заголовок 802.1q c 12 битным полем Vlan ID - тут в пакете и переносится номер Vlan-а. Т.к. поле Vlan ID - 12 бит, то оно может принимать значения от 0 до 4096 - значит не может быть Vlan-а с номером, к примеру, 4097.

|  6 байт	|  6 байт	|2байт| 4 бита | 12 бит |2байт|		|  4	|
|-----------------------------------------------------------------------|
|MAC Destin	|MAC Source	|0x8100|PCP/DEI| Vlan ID|Type | Payload	| FCS	|

Самая первая настройка, которая делается на порту свитча - выставляем switchport mode. Бывает 2-а режима access и trunk.

В режиме switchport mode access - порт отправляет/принимает только пакеты в формате Untagged.
interface Fa0/10
	switchport mode access
	switchport access vlan 22 - порт работает только с Untagged пакетами, относящиеся к потоку с vlan 22.

В режиме switchport mode trunk - порт отправляет/принимает только пакеты в формате Tagged.
interface Fa0/22
	switchport mode trunk
	switchport trunk allowed vlan 22,33 - теперь это порт сможет передавать Tagged пакеты относящиеся, только к 22 и 33 vlan-ам

Best practice:
Порты на свитче, к которым подключаются компы и сервера - всегда в режиме access.
Порты на свитче, к которым подключены другие свитчи, маршрутизаторы, firewall - всегда в режиме trunk.


3. Spanning-tree
	Задача:	Обнаружить и заблокировать петлю, т.к. у Ethernet при образовании петли сносит крышу.
	Решение:	Система строит дерево. На каждом свитче настраиваем параметр priority (0..65535). У кого priority наименьший тот ROOT. Далее ROOT во все свои линки отправляет пакеты BPDU. Если вдруг эти пакеты друг с другом пересеклись - петля, и свитчи этот линк блокируют.

Настройки Spanning-tree:
Cisco поддерживает несколько режимов (mode) работы STP: MST - используется зачастую только в провайдерских сетях, PVST+ - используется если в сети зоопарк из вендоров, т.е. есть и Juniper и Huawei, Rapid PVST+ - если сеть только из Cisco
#spanning-tree mode rapid-pvst
На свитчах настроить priority для выбора ROOT.
spanning-tree vlan 1-4094 priority 0		- значит этот свитч будет ROOT
Фактически всё, остальное тюнинг.

4. OSPF динамическая маршрутизация.
	Станция м.Таганская знает что у неё 6 линков и соответственно 6 соседних станций. Этот блок информации в OSPF называется LSA.
		LSA
	Router ID: Таганская
	Neighbor 1: Третьяковская
	Neighbor 2: Павелецкая
	Neighbor 3: ...
Т.е. у каждого роутера есть свой LSA, описывающий его линки. Теперь задача дружно обменяться этими LSA. Eсли в нашей сети 100 роутеров, то в базе на каждом из них после обмена должно оказаться = 100 LSA. Дальше на основании этих данных любой роутер, запустив "Алгоритм Дейкстры", может определить - если ко мне пришёл пакет с Destination IP 10.15.23.66, в какой из 6 своих линков я его должен засунуть, чтобы он дошёл до места назначения за минимальное количество hop-ов? В 4-ый линк. Отправил. Пакет пришёл на следующий роутер. Тот имеет такую же базу LSA, тот же "Алгоритм Дейкстры" и ...

Минимальная настройка OSPF:
interface fa0/12
	ip address 10.0.0.12 255.255.255.0
interface fa0/18
	ip address 192.168.0.54 255.255.255.0
router ospf 1 					- запустили сам deamon на устройстве
network 10.0.0.12 0.0.0.0 area 0 	- включаем ospf на интерфейсе fa0/12. Теперь OSPF будет искать neighbor только за этим интерфейсом. LSA будет содержать данные только для этого интерфейса. Про fa0/18 ничего знать не будет и передавать информацию о нём тоже не будет.


Топология.
Существую всего три топологии построения сетей:
1. Шина - пример: к единственной жиле коаксиального кабеля подключаются абоненты. Сейчас используется в основном внутри микропроцессоров.
2. Кольцо - используется провайдерами когда надо покрыть большую географическую площадь с низкой плотностью абонентов + кольцо автоматически создаёт отказоустойчивость.
3. Звезда - в одном месте куча компов. Поэтому в Enterprise в основном используется именно звезда.

В Cisco такой дизайн сети назывался Cisco Safe.
https://www.researchgate.net/profile/Rafidah_Md_Noor/publication/233815226/figure/fig1/AS:300120533946372@1448565719517/High-Level-View-of-Cisco-SAFE-Architecture-Oppenheimer-2011.png

Как строится "Звезда".

Campus - пользователи.
В центре сети (уровень Core) устанавливается самый производительный свитч. В идеале Catalyst 6500. В Гелиосе это был стек из 2-х Cisco 3750, который стоял в серверной. Представим что зданий, где сидел персонал было бы штук 5. В каждом таком здании ставим свитч классом ниже, условно Cisco 3560 (это уровень Distribution) - двумя линками соединяем с Core. Далее каждый этажный свитч (уровень Access) подключаем к уровеню Distribution.

Data Center.
Все линки от серверов подключаем либо непосредственно к Core (как вроде бы это было в Гелиосе), либо так же как и Campus через промежуточный Distribution уровень только уже для Data Center.

Internet Edge.
Блок подключения к Интернет. Состоит из Firewall и роутера, в который подключаем линк от провайдера. Блок так же одной ногой подключен к Core.

Бывают ещё блоки Management - состоит из серверов мониторинга и управления. Всякие Syslog, Cisco Works, Radius/Tacacs и т.п. Так же одной ногой к Core.

WAN Edge.
Обычно выделенные каналы между офисами. Надо офисы в Москве и Ижевске связать IP телефонией. Пробрасывать IP телефонию через Интернет - не хорошо, возможен packet lost, задержки, jitter. Заказываем у провайдера канал точка-точка с характеристиками packet lost = 0, delay < 150 ms, jitter < 5 ms.

ГПБ
Так как офис планировался небольшой (~80 человек, ~10 серверов) решили уровни Core, Campus-Access и Data Center-Access объединить в один блок: 4-е Cisco 3750 объединённые в стек.
2-е из них по 24 гигабитных порта - для подключения серверов.
остальные 2-е по 48 100 Мбит портов c PoE для подключения пользователей.
(Гигабитные и 100 мегабитные железки МОЖНО объединять в один стек!)
и в принципе этот подход спокойно расширяется, если увеличивается количество пользователей/серверов, т.е. в к стеку можно было бы подключить ещё одну 3750. (Максимум в стеке может быть 8 устройств)

Модуль Internet Edge состоял из ASA + роутер Cisco 2821.

Задача роутера
Одной ногой подключение к провайдеру, другой к ASA
+ делать NAT
+ так называемые anti-spoofing list - это типа от провайдера/из Интернета не могут прилетать пакеты с Source IP адресами 192.168.0.0, 10.0.0.0, 172.16.0.0.
+ потом у ГПБ открылся ещё один офис и надо было соединить туннелем эти офисы - на роутере поднимался этот туннель.

Задача ASA
Одной ногой в Core, второй в роутер, третьей в DMZ свитч Cisco 2960, к которому подключался FTP сервер, FrontEnd Exchange, Proxy
+ детализация access-list-ов (если на роутере большие блоки anti-spoofing list, то здесь конкретно какие IP адреса разрешаем, какие порты разрешаем)
+ все виды штфрования. Мы использовали шифрование для удалённого доступа Cisco VPN Client.
