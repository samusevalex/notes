Основные вещи, из которых состоит linux.
1. Загрузчик - grub, UEFI
2. Ядро - kernel
3. Cистема инициализации - SysV или Systemd
4. Оболочка для командной строки - Bash
5. Система управления пакетами - dpkg/apt (для Debian ветки) или rpm/yum (для Red Hat ветки).
6. Модуль управления драйверами - /lib/systemd/systemd-udevd
7. Логирование - rsyslog или /lib/systemd/systemd-journald
8. Планировщик задач - cron или systemd.timer
9. Файловая система /proc и /sys
10. Полезные tool-зы: top, vmstat, lsof, kill

Сначала ядро - kernel.
Раньше ~1950 годах на одном железе запускали одну программу. Она полностью распоряжалась процессором, памятью, устройствами ввода/вывода. Но потом решили на одном железе одновременно запускать 2 и более программ -> эти программы начинают конкурировать за ресурсы (CPU, MEM, I/O) и "арбитром" выступает Операционная Cистема.

Задачи OS:
1. Решить какая из программ в данный момент времени получает доступ к CPU -> планировщик процессов или scheduling policy. Соответственно у каждого процесса есть свой статус: 
https://www.intuit.ru/EDI/22_02_19_2/1550787571-32127/tutorial/901/objects/8/files/8_1.png
2. OS должна обладать механизмами управления памятью -> каждой программе должно выделяться отдельное место в ОЗУ.
3. Решить какая из программ в данный момент времени получает доступ к I/O - клавиатуре, hdd и т.п. -> процессы не могут напрямую взаимодействовать с внешними устройствами типа хардом. Всё взаимодействие только с помощью системных вызовов - syscalls.
4. Один процесс хочет передать другому процессу какую-то инфу -> OS предоставляет такую возможность через вещи с обобщённым названием IPC (inter-process communication) межпроцессорное взаимодействие -> это socket, pipe, shared memory...
5. И плюс ко всему OS должна понимать "внешние" устройства. Т.е. работать с драйверами.

Так вот kernel - это то, что выполняет все эти функции.
Исходник kernel лежат на http://www.kernel.org
Распакованный архив занимает ~600 Мб из них 450 Мб - драйвера + документация. И ещё 100 Мб - "драйвера" непосредственно для различных типов CPU (x86, ARM, MIPS и т.д.).
Каждая "сборка" Linux-ов будь то Debian, Rat Hat или CentOS запускают "конфигуратор" ядра
http://linuxsam.org.ua/wp-content/uploads/2016/11/Fig1-menuconfig-xfterm-696x450.jpg
из этого архива берут только нужные по их мнению модули/файлы. Далее компилируют и получают, к примеру, у меня для Debian 9:
/boot/config-4.9.0-8-amd64 - это файл с параметрами, которые мы нащёлкали в "конфигураторе" ядра
/boot/vmlinuz-4.9.0-8-amd64 - это само ядро/kernel размера ~4 Мб. Фактически это один большой exe-шник. Как видно, взят из исходника версии 4.9. К примеру, Rat Hat 6 использует версию исходника kernel 2.6.32. Т.е. все "сборки" linux сидят на www.kernel.org только разных версиях и по разному сконфигурированных.

Grub.
Процесс разгрузки Linux.
Включаем компьютер - начинает отрабатывать BIOS. BIOS смотрит в свои настройки и выбирает с какого диска будет грузиться. С этого диска берёт первый блок =512 байтам, загружает его в память и запускает его как exe-шник. В этом exe-шнике первые 446 байт - boot loader, а последние 64 байта - таблица MBR. Задача самого boot loader, загрузить в ОЗУ блоки с 2 по 2047 - где лежит программа GRUB или Lilo или ещё какой-нибудь загрузчик и запустить эту программу. Далее GRUB в своём коде имеет понимание как работать с файловыми системами, лезет на диск в папки, смотрит свой .conf файл, понимает где лежит kernel, полностью загружает его в ОЗУ и запускает с параметрами, к примеру: /boot/vmlinuz-4.9.0-8-amd64 init=/sbin/init
init=/sbin/init - говорит о том запустится файл /boot/vmlinuz-4.9.0-8-amd64, который в свою очередь запустит exe-шник /sbin/init

/sbin/init
Это первый запускаемый процесс в системе linux - Cистема инициализации. Что-то типа Автозагрузки в Windows. Изначально задача была такой: при загрузке OS запускать программы типа deamon собирающего логи или deamon, который отслеживает подключение новых устройств к машине. 
В истории систем инициализации было несколько.
1-ая была Upstart - сейчас практически нигде не используется.
Потом появилась SysV = initd = /sbin/init
Сейчас практически все переходят на Systemd.

С SysV я не работал, там в качестве .conf файлов используется /etc/rc0.d, в которых в чистом виде написаны bash скрипты, типа

#!/bin/bash
/sbin/dhclient 
Что будет означать, запустить exe-шник dhclient, который по DHCP будет получать IP.

В systemd если надо создать сервис, который будет постоянно работать в фоне пишем файлик типа

# /lib/systemd/system/cron.service
[Unit]
Description=Regular background program processing daemon
Documentation=man:cron(8)

[Service]
EnvironmentFile=-/etc/default/cron
ExecStart=/usr/sbin/cron -f $EXTRA_OPTS
IgnoreSIGPIPE=false
KillMode=process

[Install]
WantedBy=multi-user.target

Таким образом systemd или SysV помимо всего остльного запускают программку /sbin/agetty - которая позволяет подключаться серверу по ssh. Когда пользователь подключается по ssh, /sbin/agetty запускает /bin/login, если это SysV или /lib/systemd/systemd-logind если это Systemd. Login выдаёт приглашение, типа "Привет, Вася", которое лежит в файле /etc/issue, потом проверят пользователя на пароль и если всё хорошо запускает /bin/bash, которое пользователь и видит при заходе на терминал. Далее всё что не введёт пользователь в консоле, будет интерпретироваться этим exe-шником /bin/bash.

Bash = PowerShell думаю ничем они друг от друга не отличаются.

Установка программ в Linux.
Я работаю с Debian, поэтому у меня пакетный менеджер - dpkg и apt, а файлы с расширением .deb Для Rat Hat это будет тоже самое, только rpm/yum/.rpm
Самое неправильное - это скачать какую-нибудь кривую программу в исходных кодах из интернета, начать её скомпилировать. Она создаст в разных папках какие-то temp файлы, куда-нибудь запишет какие-нибудь библиотеки. А потом долго придётся такую прогу удалять.
Поэтому в Linux, используется пакетный менеджер и репозитории.
По феншую АБСОЛЮТНО все программы скачиваются как .deb файлы - это zip архив со своей структурой каталогов, в которых лежат помимо самой программы ещё и скрипты для установки/удаления + ещё какая-нибудь meta информация. Скачав файл example1.deb с сайта производителя, запускаем dpkg --install example.deb и программа установится, т.е. распакуется, проверит все ли библиотеки есть на компе для её работы, а главное сам dpkg будет знать где какие файлы расположены.
В Linux, как и в Android или IOS есть ещё подход с использованием репозиториев. Т.е. на авторитетном сайте создаётся большая файловая шара,
в которой лежат программы точно работающие без багов под данную систему. К примеру, для Debian есть репозиторий на mirror.yandex.ru. Я этот сайт прописываю в файле /etc/apt/sources.list, а потом в командной строке пишу apt install example1.
